    What is Lighthouse


    Lighthouse is developed by engineers for engineers providing common resource of useful commands and support links to help in troubleshooting and knowledge colaboration.

    This portal is developed for engineers to edit, add and delete resources. Once submitted, the new content will be reviewed by admins to validate and check accuracy before it becomes available to all users.

    The resources (commands and links) are grouped by Collection/Platform/feature.

    Commands section is intended to provide helpful commands and comments as a tool for troubleshooting.

    Links section is intended to provide helpful links and or BU alias for further investigation. Helpful resources troubleshooting could be tech-zone, Cisco Wikis, webex recording, topic or any Cisco documents.

    Please check the "Guidlines" section below to see how the new content should be formatted


    General rules and guidelines


    When entering new content, it’s very important to follow the below rules so your data becomes useful and presentable to all engineers.

    • Make sure the commands are relevant and useful in troubleshooting a certain platform/feature
    • Make sure the syntax is correct and the commands actually work on the platform intended.
    • When possible, please add a description/comment/instruction to commands to explain or add instruction or comment. (please check the formatting guidelines)
    • Notes/comments should be kept short, simple and easy to read.
    • The commands list shall be simplified and easy to read.
    • If the list is too long, please group and divide into sections if possible.
      1. Grouping is accomplished by grouping commands according to hierarchy, module and function or type or issue. (please check the formatting guidelines for more details)
      2. Dividing into sections could be accomplished by adding headers. (please check the formatting guidelines for more details)
    • Verify that the links and aliases included in the links section are working and correct before adding.
    • Use the feedback email at top right for any questions or suggestion related to this tool

    Formatting guidelines


    For successful entry of content and best presentation on Lightning, please follow those guild lines in the following areas:

    Collections:

    • To have a new Collection send your request to the Lighthouse mailer. Describing the area to be covered and potential underlying platforms and feature.

    Platform and feature:

    • When creating a new platform or new feature, please make sure that there’s no spaces and no special characters in the string and replace by the minus symbol ‘-‘ if needed.

    Commands section:

    • Variables shall always be in square brackets [ ]
      • Example: show logging location [node-id]
    • To add notes/description/comment or instruction, use double forward slashes ‘//’ characters prior to a description/comment and include the description/comment in parentheses ‘( )’
      • Example: show interface [int-type][int-number] //(look for drops incrementing)
    • Grouping
      • (Ex:by hierarchy):
         show controllers fabric fia
         show controllers fabric fia link-status location []
         show controllers fabric fia stats
         show controllers fabric fia stats location []
         show controllers fabric fia drops [ ingress | egress ] loc[]
    • Dividing into sections could be accomplished to group certain collection of commands together by writing a title and using the colon symbol after the title/header:
      • Example:
          show techs:
           show tech-support file [media-id]:[file-name]
           show tech routing bgp
           show tech multicast
          traces:
           show tcp trace
           show ... trace

    Links section:

    • A header to the link is recommended to give a brief explanation of what the link contains.
    • The header shall come prior to the link and separated by ’ >’ sign.
      • Example:
         Wiki - Fabric Driver > https://wiki.cisco.com/display/GSRIN/Fabric+Driver
    • Email alias can be added. The ‘@’ sign has to be included for the system to recognize that it’s an email address and format correctly.
      • Example:
         cs-ncs5500@cisco.com
